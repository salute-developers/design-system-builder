# Создание и добавление компонента в конструктор дизай-системы

## I / Локальное развёртывание сервиса

Для запуска сервиса локально, необходимо наличие Docker приложения на компьютере и любой клиент, который поддерживает работу с ним.

Требуется склонировать репозиторий себе и открыть его через любой редактор кода.

```
$ git clone https://github.com/salute-developers/design-system-builder.git
```
Далее необходимо выполнить скрипт, который развернёт все необходимые сервисы.

```
$ ./setup-docker.sh
```
Затем перейти в директорию

```
$ cd services/db-service
```
И выполнить команду для насыщения базой данных “продовыми” компонентами. При этом перетрутся все тестовые данные.

```
$ npm run db:seed:prod
```

## II / Добавление компонента

На странице [http://localhost:3004](http://localhost:3004) /settings в разделе “Components” необходимо добавить компонент, указав его название (так же как оно написано в ядре) и описание.

## III / Добавление пропсов компонента

В примере указанного файла лежат все пропсы компонента. Необходимо добавить их в админской панели на вкладке “Components” с соответствующими типами:

Пример токена | Тип | Пример значения
|---|---|---|
background | color  | text.default.primary
borderRadius | shape | round.xl
height | dimension | 40
focusAlpha | float | 0. 5
labelStyle | typography | body.l.normal

Пример для компонента “Counter”:

https://github.com/salute-developers/plasma/blob/dev/packages/plasma-new-hope/src/components/Counter/Counter.tokens.ts

```
В будущем добавление элиасов нужно будет делать не только для веба, но и для нативных платформ
```
## IV / Добавление вариаций

Вариации - это набор свойств компонента, которые влияют на его отображение. Для поиска поддерживаемых вариаций необходимо понять, какие вариации есть в ярде.

Например, для компонента “Counter” можно найти эту информацию в конфигурационных файлах https://github.com/salute-developers/plasma/blob/dev/packages/plasma-new-hope/src/examples/components/Counter/Counter.config.ts в поле `variations`, а именно: 'view' и 'size'.

Затем, на странице [http://localhost:3004](http://localhost:3004) /settings в разделе Variations необходимо добавить эти вариации с описанием:

Вариация | Описание
|---|---|
Size | Размер
View | Вид
Shape | Форма

```
В будущем описание не будет строго требоваться
```

## V / Установка ассоциаций пропсов с вариациями

Чтобы у конечного пользователя была возможность добавлять стили для
вариаций, необходимо установить ассоциации пропсов с вариациями. Для
этого следует просмотреть тот же файл с конфигурацией, понять, какие
пропсы используются для каждой вариации и проставить эти соответствия
через админскую панель на вкладке Prop → Variation. Для ориентира можно
брать любой стиль.

Так же существуют пропсы, которые не относятся ни к одной вариации и
применяются не зависимо от включенных свойств. Такие пропсы называются
инвариантами и он помечены как “NOT LINKED TO ANY VARIATION”

## VII / Добавление сторисов

Чтобы компонент рендерился, и чтобы к нему можно было добавить вариации, указать значение пропсов и т.д., необходимо добавить код компонента.

Как правило, пример кода уже есть в кодовой базе ядра, достаточно его скопировать и адаптировать под структуру.

Например то, как используется компонент “Counter", можно посмотреть тут https://github.com/salute-developers/plasma/blob/dev/packages/plasma-new-hope/src/examples/components/Counter/Counter.stories.tsx. 

И после добавление стори необходимо “зарегистрировать” её в двух файлах:

```
В apps/client/src/stories/index.ts
В apps/client/src/hooks/useStory.ts
```
```
Возможно в будущем “регистрацию” файлов можно будет не делать
```

## VIII / Создание базовых стилей для вариаций

Когда компонент везде добавлен, необходимо создать ему стили для каждой вариации, и после чего расставить корректные значения для базовой дизайн системы.
Сами значения можно взять в базовой библиотеки компонент в фигме, или в конфиге компонента библиотеки sdds-serv.

Перед тем как начать, нужно авторизоваться под админской учётной записью (user = admin, pass = plasma-ds), которая отобразит только одну **БАЗОВУЮ** дизайн систему, в которой и нужно создавать и наполнять все компоненты.

Например, для компонента “Counter” стили view=ʼdefaultʼ и size=ʼlʼ эти значения можно найти в конфигах (из раздела IV)

Тогда по аналогии нужно через интерфейс конструктора дизайн систем перенести эти значения туда.


### ВАЖНО

```
Eсли значение в конфигах указано в rem, то нужно умножать его на 16. Не забыть установить значения по умолчанию для стилей.
```

## IX / Генерация готового компонента в прод сида

Когда компонент полностью оформлен, необходимо дополнить продовый сид, для переноса этих данных в облачную инфраструктуру.

Для этого нужно перейти в следующую директорию

```
$ cd services/db-service
```

И вызвать команду и вместе с флагом имя компонента

```
$ npm run db:seed-generate:prod -- --component="<component_name>"
```

После чего, изменения нужно будет закоммитить и создать пул-реквест на влитие.

## X / Добавление компонента в прод

Завершающим шагом является перенос данных созданного компонента в продовую среду, которая находится в облачной инфраструктуре.

Для этого нужно перейти в следующую директорию

```
$ cd services/db-service
```
Поменять .env файл на актуальный адрес базы данных

```
...
DATABASE_URL=postgresql://postgres:postgres@localhost:5433/ds_registry
....
```
И вызвать команду и вместе с флагом имя компонента

```
$ npm run db:seed:prod -- --component="<component_name>"
```

После чего компонент добавится в базу данных на проде, и нужно будет в ручную проставить во все нужные дизайн системы на странице админской панели https://admin design-system-builder.ru/settings с помощью вкладки ”Design Systems”.
